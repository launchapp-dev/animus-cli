Command Help + Argument Clarity

CLI Help / Task Update / Input contracts

ready format-check execute

$ ao task update --help

update - Update an existing task using explicit flags or --input-json

Usage:
  ao task update --id <TASK_ID> [OPTIONS]

Required:
  --id <TASK_ID>           Task identifier (for example: TASK-002)

Options:
  --status <STATUS>        Task status. Accepted values:
                           backlog|todo, ready, in-progress|in_progress, blocked,
                           on-hold|on_hold, done, cancelled
  --type <TYPE>            Task type. Accepted values:
                           feature, bugfix, hotfix, refactor, docs, test, chore, experiment
  --priority <PRIORITY>    Task priority: critical, high, medium, low
  --deadline <YYYY-MM-DD>  Deadline in ISO date format
  --input-json <PATH>      Load update payload from JSON file; when set, JSON values
                           take precedence over individual field flags
  --json                   Emit ao.cli.v1 envelope
  -h, --help               Print help

Examples:
  ao task update --id TASK-002 --status in-progress --priority high
  ao task update --id TASK-002 --input-json ./task-update.json --json

Argument guidance checklist

  • Accepted values shown in deterministic order.
  • Alias forms (`in-progress|in_progress`) stay explicit.
  • `--input-json` precedence is called out in-line.
  • Examples include both human and `--json` automation paths.

Expected operator result

A valid command can be assembled without source-code lookup and rerun once if validation fails.

Invalid Value Recovery + JSON Parity

CLI Errors / Parse Validation / One-rerun recovery

error-shown corrected-rerun

$ ao task update --id TASK-002 --status paused

error: invalid status 'paused'; expected one of: backlog, todo, ready, in-progress, in_progress, blocked, on-hold, on_hold, done, cancelled; run 'ao task update --help'

rerun:
  ao task update --id TASK-002 --status in-progress

$ ao task update --id TASK-002 --status paused --json

{
  "schema": "ao.cli.v1",
  "ok": false,
  "error": {
    "code": "invalid_input",
    "message": "invalid status 'paused'; expected one of: backlog, todo, ready, in-progress, in_progress, blocked, on-hold, on_hold, done, cancelled; run 'ao task update --help'",
    "exit_code": 2
  }
}

Deterministic recovery contract

  • Error always includes domain, invalid token, accepted values, and rerun hint.
  • Clause order and punctuation stay stable for snapshot tests.
  • Human and JSON mode carry the same corrective meaning.

Confirmation Required + Dry Run Preview

Destructive flow / Safety gate / Preview then confirm

confirmation-blocked preview-rendered confirmed-rerun

$ ao git worktree remove --repo ao-cli --worktree-name task-task-002

error: CONFIRMATION_REQUIRED: rerun 'ao git worktree remove --repo ao-cli --worktree-name task-task-002' with --confirmation-id CONF-7F3A; use --dry-run to preview changes

preview command:
  ao git worktree remove --repo ao-cli --worktree-name task-task-002 --dry-run

task/workflow variant:
  CONFIRMATION_REQUIRED: rerun 'ao workflow cancel --id WF-42' with --confirm WF-42; use --dry-run to preview changes

$ ao git worktree remove --repo ao-cli --worktree-name task-task-002 --dry-run --json

{
  "schema": "ao.cli.v1",
  "ok": true,
  "data": {
    "operation": "git.worktree.remove",
    "target": "task-task-002",
    "destructive": true,
    "dry_run": true,
    "requires_confirmation": true,
    "planned_effects": [
      "validate worktree path exists",
      "verify branch and worktree mapping",
      "remove worktree and branch on confirmed rerun"
    ],
    "next_step": "rerun with --confirmation-id CONF-7F3A"
  }
}

Safety guidance consistency

  • `CONFIRMATION_REQUIRED` token appears unchanged across groups.
  • Exact confirmation flag name is present in every blocking message.
  • `--confirmation-id <id>` guidance appears for request-based approvals.
  • `--confirm <token>` guidance remains explicit for task/workflow controls.
  • `--dry-run` path is always promoted before mutation.
  • Dry-run payload keeps shared keys in stable top-level order.
