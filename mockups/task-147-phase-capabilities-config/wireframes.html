S2 Capability Field Definitions

Five capability fields replace all existing boolean category functions. Each field maps one-to-one to a decision axis currently scattered across multiple Rust files.

can_write

bool

Controls write action rule injected into phase prompt. Replaces is_write_phase() in phase_executor.rs line 202.

needs_commit

bool

Requests commit_message in output contract. Replaces phase_requires_commit_message() in phase_executor.rs line 289.

model_group

"ui_ux" | "research" | "review" | "requirements" | "testing" | "default"

Drives primary model selection. Replaces the five is_*_phase() branches in model_routing.rs and the two in phase_targets.rs.

parse_research_signal

bool

Whether to scan stdout for {"{"}"kind":"research_required"{"}"} during execution. Currently implicit for the research group.

safety_rules_template

string | null

Named safety rule block injected into prompt. "research" emits the greenfield/targeted-discovery paragraph currently hardcoded in phase_safety_rules() line 265.

S3 Config Schema Authoring

phase_capabilities in agent-runtime-config.v2.json

The phase_capabilities key is added to each phase entry alongside the existing mode, directive, and runtime fields. Omitting it falls back to compiled defaults, preserving backward compatibility during migration. 

// agent-runtime-config.v2.json (proposed)
{
  "phases": {
    "wireframe": {
      "mode": "agent",
      "agent_id": "default",
      "directive": "Create UI mockups...",
      // NEW — replaces 5 Rust match arms
      "phase_capabilities": {
        "can_write": true,
        "needs_commit": false,
        "model_group": "ui_ux",
        "parse_research_signal": false,
        "safety_rules_template": null
      },
      "runtime": null,
      "output_contract": null
    }
  }
}

Rust Code Before → After

Concrete code change: the three separate boolean functions collapse into a single config read. 

// BEFORE — phase_executor.rs (line 202)
let is_write_phase = matches!(
    phase_id,
    "implementation" | "wireframe" | "design"
);

// BEFORE — model_routing.rs (line 169)
fn is_ui_ux_phase(phase_id: &str) -> bool {
    matches!(phase_id,
        "ux-research" | "wireframe" | "mockup-review"
        | "design" | "ui-design" | "ux-design"
    )
}

// AFTER — both callers
let caps = config.phase_capabilities(phase_id);
let is_write_phase = caps.can_write;
let model_group   = caps.model_group; // "ui_ux"

S5 Config-Driven Routing Flow (After)

Instead of branching on phase_id strings in three Rust files, the daemon reads phase_capabilities from config at startup. Each decision axis becomes a single field lookup.

1. Phase starts

phase_id = "wireframe"

2. Load config

agent-runtime-config.v2.json
phases["wireframe"]

3. Read capabilities

model_group = "ui_ux"
can_write = true
needs_commit = false

4. Route model

primary = "gemini-3.1-pro-preview"
(via model_group)

5. Set prompt rule

can_write → write action rule
needs_commit → false

Before: 3 separate match arms in 3 files — is_ui_ux_phase() (6 ids), is_ui_ux_phase() (5 ids — different set!), is_write_phase() (3 ids). After: one config read, one field lookup per decision axis. Adding a new phase group requires only a config edit.
